## Правила работы с репозиторием
1. Работа с ветками (Git Flow Light)
   Используем упрощенную модель Git Flow. Основная ветка — main (или master). Любое изменение кода происходит через Pull Request (PR).
   Именование веток
   Формат: тип/номер-задачи_краткое-описание
* feature/ — для нового функционала.
    * Пример: feature/HOCH-38AuthMiddleware
* bugfix/ — для исправления ошибок.
    * Пример: bugfix/HOCH-38FixNilPinter
* hotfix/ — срочные правки на продакшн (идут в обход обычного цикла).
    * Пример: hotfix/HOCH-38criticalSecurityPatch
      Требования к PR
* Small PRs: Один PR — одна задача. Если кода больше 400 строк, его нужно разбивать.
* Linter & Tests: PR не принимается, если не проходят тесты или линтер (golangci-lint).
* Review: Минимум 1 одобрение (approve) от другого разработчика или тимлида.
3. Семантическое версионирование (SemVer)
   Мы используем формат vX.Y.Z:
* X (Major): Обратно несовместимые изменения API.
* Y (Minor): Новый функционал с сохранением совместимости.
* Z (Patch): Багфиксы и мелкие правки.
  Работа с тегами
* Тег ставится только на ветку main после слияния PR.
* Имя тега должно соответствовать версии: v1.2.3.
* Для каждого сервиса в монорепозитории (если используете) тег включает имя сервиса: auth/v1.0.2.
3. Пайплайн реализации фичи
   Пошаговый алгоритм для джуна:
* Обсуждение: Перед кодингом проговорить архитектурное решение с тимлидом (5-10 мин).
* Ветка: Создать ветку от актуального main.
* Локальная разработка:
    * Код должен соответствовать Uber Go Style Guide.
    * Написание Unit-тестов обязано покрывать основной сценарий.
* Сборка: Проверка, что go mod tidy выполнен и проект компилируется.
* Push & PR: Создание Draft PR для раннего фидбека или обычного PR для ревью.
4. Пайплайн деплоя
   Процесс автоматизирован через CI/CD (например, GitLab CI или GitHub Actions).
   Сценарий A: Фичи и обычные багфиксы
* Merge в main: После аппрува ветка вливается в main.
* Staging Deploy: Автоматический деплой в тестовую среду (Staging/Dev).
* QA: Проверка функционала в окружении.
* Tagging: Создание Git-тега (например, v1.1.0).
* Production Deploy: После создания тега запускается пайплайн деплоя на прод.
  Сценарий B: Hotfix (Критический баг)
* Создается ветка hotfix/ от последнего стабильного тега или main.
* Исправление + тест.
* Срочный PR и деплой на Staging.
* Слияние в main и немедленный выпуск патч-версии (v1.1.1).
5. Специфика Go в команде
* Обработка ошибок: Никаких «пустых» if err != nil { return err } без контекста. Использовать fmt.Errorf("context: %w", err).
* Линтеры: Обязательное использование golangci-lint с включенными govet, errcheck, staticcheck.
* Логи: Использовать структурированное логирование (например, slog или zap).



# Кастомный логгер на основе slog

В приложении используется стандартный логгер `log/slog` с собственной инициализацией и поддержкой контекста.

---

## 1. Глобальный логгер `slog.Default()`

Логгер настраивается при старте приложения на основе .env и устанавливается как глобальный по умолчанию в main.go.

После этого в любом месте приложения можно вызывать **глобальный кастомный** логгер.

**Примеры вызовов:**

```go
slog.Default().Info("message")   // Default() возвращает установленный логгер
slog.Default().Error("failed", "err", err)
```

**Когда использовать глобальный логгер:**

- В коде, где нет `context` или не нужна привязка к запросу (request_id и т.п.).

**Когда не использовать:**

- Внутри HTTP-обработчиков и сервисов, где уже есть контекст запроса — там лучше логгер из контекста (см. раздел 3).

---

## 2. Уровни логирования

В `slog` четыре стандартных уровня (от самого «тихого» к самому «громкому»):

| Уровень | Значение | Когда использовать | Метод |
|--------|----------|--------------------|--------|
| **Debug** | -4 | Детальная отладочная информация: вход/выход функций, промежуточные значения, зарегистрированные маршруты. В проде обычно отключён. | `log.Debug(msg, key1, val1, ...)` |
| **Info** | 0 | Обычный ход работы: старт приложения, входящий запрос, успешные важные действия. | `log.Info(msg, key1, val1, ...)` |
| **Warn** | 4 | Потенциальные проблемы: повторные попытки, устаревший API, деградация без падения. | `log.Warn(msg, key1, val1, ...)` |
| **Error** | 8 | Ошибки: сбой операции, исключительная ситуация. Всегда передавайте ошибку. | `log.Error(msg, key1, val1, ...)` |

Уровень приложения задаётся в конфиге .env. Сообщения с уровнем **ниже** установленного отбрасываются. Например, при `info` не выводятся записи уровня `Debug`.

**Примеры вызовов:**

```go
log := slog.Default()

log.Debug("execution time", "key", Value, "Time", 60)
log.Info("server started", "addr", addrValue)
log.Warn("retry attempt", "attempt", n, "err", err)
log.Error("database connection failed", "err", err)
```

В качестве err можно передавать кастомную ошибку типа TraceError.
Первым идёт сообщение лога, а далее параметры передаются парами «ключ — значение»; количество пар произвольное.

## 3. Логгер в паре с контекстом

Чтобы все логи в рамках одного запроса имели общие поля (например `request_id`), используется **кастомный логгер в контексте**.

### 3.1. Создать кастомный логгер и положить в контекст

Где угодно можно взять текущий логгер (глобальный или из контекста), добавить поля через `.With()` и сохранить логгер в контекст:

```go
customLog := slog.Default().With(
    "request_id", requestID,
    "method", r.Method,
    "path", r.URL.Path,
)
ctx = logger.LoggerToContext(ctx, customLog)
```

Функция **`LoggerToContext(ctx, l)`** из пакета `logger` добавляет логгер в контекст и возвращает новый контекст. Дальше этот контекст нужно передавать по цепочке вызовов (например, `r.WithContext(ctx)` в HTTP).

### 3.2. Достать логгер из контекста и залогировать

В любом месте, где есть этот контекст:

```go
log := logger.LoggerFromContext(ctx)
log.Info("processing request")
log.Error("handler failed", "err", err)
```

Если в контексте логгера не было (например, вызвали код без middleware), **`LoggerFromContext`** вернёт `slog.Default()` — логи всё равно попадут в глобальный логгер.

### 3.3. Пример в приложении

**Middleware** создаёт логгер с полями запроса и кладёт его в контекст:

```go
loggerCtx := slog.Default().With(
    "request_id", requestID,
    "method", r.Method,
    "path", r.URL.Path,
    "remote_addr", r.RemoteAddr,
    "user_agent", r.UserAgent(),
)
loggerCtx.Info("→ incoming request")

ctx := logger.LoggerToContext(r.Context(), loggerCtx)
r = r.WithContext(ctx)
```

**Обработчик или сервис** достаёт логгер и пишет логи с тем же `request_id`:

```go
l := logger.LoggerFromContext(r.Context())
l.Info("LivenessProbe called")
l.Error("operation failed", "err", err)
```

Так все записи по одному запросу можно отфильтровать по `request_id` в логах.

---

## Краткая шпаргалка

| Задача                                      | Что использовать |
|---------------------------------------------|-------------------|
| Лог в main / без контекста                  | `slog.Default().Info(...)` или `slog.Info(...)` |
| Уровни                                      | `Debug` — отладка, `Info` — норма, `Warn` — предупреждения, `Error` — ошибки |
| Передать ошибку                             | `log.Error("msg", "err", err)` |
| Кастомнный логгер с доп. полями в контексте | `l := slog.Default().With(...)` → `ctx = logger.LoggerToContext(ctx, l)` |
| Залогировать в обработчике/сервисе          | `log := logger.LoggerFromContext(ctx)` → `log.Info(...)` / `log.Error(...)` |

# Кастомные ошибки (TraceError)

Тип `TraceError` из собственного пакета сервиса `errors` хранит сообщение, причину (вложенную ошибку), опционально стек вызовов и произвольные поля для логов. При логировании через slog выводятся структурированные данные (сообщение, цепочка причин, стек, метаданные).

---

## 1. Создание

Функция **`errors.NewTraceError(msg string, err error, keyAndValues ...any)`** создаёт кастомную ошибку.

- **`msg`** — сообщение ошибки
- **`err`** — вызывающая ошибка, может принимать типы: `nil`, обычная ошибка (`fmt.Errorf`, ошибка из пакета и т.д.) или уже существующая `*TraceError`.
- **`keyAndValues`** — произвольные пары ключ–значение. Они попадают в структурированный лог при выводе через slog.

---

## 2. Варианты создания

### Первая в трейсе

Ошибка возникла здесь или пришла «обычной» — оборачиваем в `TraceError`, стек запишется:

- Ошибка без причины (самая нижняя в цепочке)
```go

return errors.NewTraceError("config file not found", nil)
```
- Ошибка из другой библиотеки или fmt.Errorf
```go
if err != nil {
    return errors.NewTraceError("failed to connect to DB", err)
}
```

**Важно: стек сохраняется только у первой в цепочке кастомной ошибки.**

- Если `err == nil` или `err` **не** является `*TraceError`, то для новой ошибки **сохраняется текущий стек** вызовов (место, где вызван `NewTraceError`). Это и есть «первая в трейсе» кастомная ошибка.
- Если `err` уже является `*TraceError`, стек **не** дублируется: у обёртки своего стека нет, в логах будет только стек из вложенной ошибки.

Так в цепочке будет один стек — в месте первого перехвата ошибки.

### Оборачивание уже существующей TraceError (стек не дублируется)

Добавляем контекст слоя, стек остаётся от первой кастомной ошибки:

```go
if err != nil {
    return errors.NewTraceError("create user", err)  // err уже *TraceError — свой стек не пишем
}
```

### На основе любой другой ошибки

Работает и с `*TraceError`, и с обычной ошибкой:

```go
err := repo.GetByID(ctx, id)
if err != nil {
    return errors.NewTraceError("get user by id", err)
}
```

### На основе с дополнительными данными

Метод **`AddMetaData(key string, value any)`** добавляет к ошибке произвольные пары ключ–значение.

```go
err := errors.NewTraceError("validation failed", nil, "user_id", userID, "field", "email")
```

В логе появятся поля `user_id`, `field` вместе с сообщением и стеком.

## 3. Проброс и обёртка

Кастомную ошибку нужно **либо пробрасывать наверх как есть**, **либо оборачивать в другую кастомную ошибку** через `NewTraceError`.

**Не стоит делать так:**

```go
// Плохо: оборачивание через fmt.Errorf с %w
return fmt.Errorf("SimpleError: %w", traceError)
```

При таком оборачивании **сохраняются только тексты сообщений**, а тип `*TraceError` и **стек теряются**. В логах не будет структуры (msg, cause_error, stack, metaData), только одна строка.

**Правильно:**

- Проброс ошибки без изменений

```go
if err != nil {
    return err
}
```

- Оборачивание в новую TraceError (стек уже есть у первой в цепочке)

```go
if err != nil {
    return errors.NewTraceError("create order", err)
}
```
